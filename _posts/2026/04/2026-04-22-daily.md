## 方法论

### Codex 上下文管理实战经验汇总

灵动的河马首次使用 Codex 遇到上下文满导致 413 错误的问题，引发了群友关于上下文管理的深入讨论。
冷静的麋鹿建议到 70% 时人工压缩新开任务，并给出了具体的 5 项压缩 prompt 模板（任务目标、已完成内容、改过的文件、当前阻塞、建议下一步）。还分享了一个进阶技巧：新开窗口让 agent 去读之前的 session 继续任务。
洒脱的仓鼠更保守，一般到 50% 就新开任务。坦荡的灰熊指出注意力到 200K 就开始涣散。热情的狮子引用 Opus 1M 的数据称到临近 1M 还有 72% 的 focus。
风趣的企鹅提醒读 session 有时会导致 context 瞬间又满，冷静的麋鹿建议用工具读取再用 subagent 总结来规避。洒脱的仓鼠表示自从用 Opus 1M 后 95% 以上的任务都跑不超过 60%，基本 one shot 完成。

> 冷静的麋鹿：跟他说：只输出以下5项，控制在15行内：任务目标、已完成内容、改过的文件、当前阻塞、建议下一步。不要解释，不要重复背景。然后拿着这个新开一个。

> 洒脱的仓鼠：我自从用了opus1m绝大部分活（95%+）现在都跑超不过60%，很少手动干预了

> 风趣的企鹅：最后这个办法有的时候极度不靠谱，我遇到过读完session以后瞬间context又满了的

### MCP 协议的实际痛点：协议说明比代码还长

坦荡的灰熊吐槽自己写了一个 MCP 工具，代码 1623 行，而压缩版协议说明就有 1774 行，"那还不如直接喂源码了"。洒脱的仓鼠补充了另一个常见痛点：在已有完整 API server 上再套一层 MCP，导致要同时维护两个 service；更有人在 MCP 上又套了 CLI 给 skill 用，变成 skill→cli→mcp→api 的奇怪复杂架构。开朗的海豚表示自己干过这种事，"现在已经离职了，后面谁接手谁倒霉"。

> 坦荡的灰熊：我写了一个mcp工具, 代码1623行, 而协议说明, 还是压缩版, 1774行。那还不如直接喂源码了

> 洒脱的仓鼠：还有个很常见的场景是已经有了一个完整的api server了，然后在上面又套一层mcp，这下要同时维护两个service了

> 开朗的海豚：干过的表示现在已经离职了，后面谁接手谁倒霉

### AI Adoption 推广策略：IC 视角下的务实建议

豁达的北极熊分享了在公司推动 AI adoption 时的纠结：大部分 DS 同事不知道 CLI 是什么，是先鼓励大家随便写降低门槛，还是从一开始建立 best practice？
沉稳的狐狸给出了精彩的分析框架：要区分最大化 adoption、最大化产出、最大化个人功劳归属三个目标。对于 IC/TL 身份，推 adoption 没有 organizational leverage 支撑，本质上是对政治资本的消耗。建议把自己的工具用好，找两三个同频的人做 champion case 即可，对 perf review 和向上汇报都足够。
洒脱的仓鼠补充道：跟不同频的人交流非常费劲，做好榜样就行，愿意的人会主动来，不愿意的人会天天说 AI 幻觉来 block 你。

> 沉稳的狐狸：如果是IC的话，你讲话别人也不一定要听你的。强行推的话，它本质上是一种对政治资本的消耗而不是build-up，这就很痛苦。

> 洒脱的仓鼠：是的，跟不同频的人交流是非常费劲的，可能会花了很多时间很大功夫两边的人都不喜欢你。你做好榜样就行了

> 豁达的北极熊：adoption拿来写十倍的垃圾不如不要adopt

### 给 AI 做工具的难点：缺乏视觉反馈和中断机制

坦荡的灰熊感叹给 AI 做工具比想象中难。以 TCP 通信为例：人类看到报文疯狂刷屏就知道有问题，但 AI 没有这种全局视野，也没办法触发中断。这暴露了 AI agent 在工具使用场景中缺乏实时监控和异常感知能力的根本问题。

> 坦荡的灰熊：比如TCP通信的时候, 人类如果看到突然返回报文疯狂刷屏,就会知道有问题. 但ai没这个视野，也没办法触发一个中断

### Codex 用量计算的"草台"发现
