今天群里信息量爆炸，<u>洒脱的鸳鸯</u> 与 <u>坦荡的灰熊</u>、<u>地球首帅鸭哥</u> 深入讨论了多智能体工程中 skill、subagent 与独立进程的分层架构设计；<u>冷静的麋鹿</u> 发起了 Claude 4.6 vs 4.7 的选型讨论；<u>今天群内信息量极大</u> 分享了对 Cursor Agent Harness 文章的解读，提炼出 evaluation-first 的核心思路。此外，Claude 把"eat your own dogfood"翻译成「自食其狗」引发群友一阵欢乐，<u>今天群内信息量极大</u> 和 <u>沉稳的兔子</u> 还试图对日报系统发起"注入攻击"，效果有待验证。

## 方法论

### 多智能体工程：从 Skill 到独立进程的分层架构

<u>坦荡的灰熊</u> 分享了自己在 Claude Code 中使用 skill + subagent 的实践痛点：vibe coding 出来的工具脚本遇到 bug 时需要停下来修改，浪费上下文，且多项目复用时维护困难。<u>洒脱的鸳鸯</u> 提出了更进一步的方案——在 skill 中调用脚本 fork 出独立的 agent 进程，相比 subagent 的优势在于新进程可以继续分出子进程，实现递归式任务分解（plan → implement → review 各阶段独立进程），主 agent 上下文保持干净。

> <u>洒脱的鸳鸯</u>：这就比 subagent 先进了，subagent 没办法继续分 subagent

> <u>地球首帅鸭哥</u>：对于需要长期维护的项目，我认为架构还是得自己来把握，模块的边界是需要思考的，然后把各种屎关在屋子里不要让他流出来

<u>洒脱的鸳鸯</u> 补充说曾考虑自己复刻一套 OpenCode，但发现 Claude Code 本身做得够好，最终把 scope 收窄到「稍长任务的自动推进」。<u>沉稳的兔子</u> 则提到自己打了一个 Python 包，按 role 分发 dispatch。

### Team Skills 实践中的优先级与性能问题

<u>80-HD</u> 分享了使用 Claude Code 驱动 team skills 时的发现：系统 skill 会优先于 personal skill 被触发，解决方案是直接删掉冲突的系统 skill；另外每次使用都会先读 personal skills 再读 team shared skill，明显变慢，猜测可以把所有 shared skill 显式声明在 index.md 里一次读完。

### Claude 4.6 vs 4.7 的选型体感

<u>冷静的麋鹿</u> 表示被各方声音搞迷糊了，发起投票。<u>热情的狮子</u> 和 <u>敏锐的雪豹</u> 给出了比较一致的建议：coding 长程任务用 4.7，文字调研创意类用 4.6；4.7 在 200K 以内 context 表现强，超过 60% 就不太灵，需要及时清理上下文。

> <u>敏锐的雪豹</u>：文字调研创意处理类 4.6，coding 长程任务 4.7，200k以内context 强，超过 60% 不灵

### Evaluation-First：Cursor Agent Harness 文章解读

<u>今天群内信息量极大</u> 分享了自己对 Cursor 新发布的 Agent Harness Engineering 文章的[解读](https://yage.ai/share/cursor-agent-harness-evaluation-first-20260501.html?utm_source=wechat&utm_medium=share&utm_campaign=cursor-agent-harness-evaluation-first-20260501)，认为文章中 best practice 本身群里很多人已在实践，真正值得深挖的暗线是 evaluation-first mindset。

### Embedding 模型选择

<u>沉稳的兔子</u> 询问 BGE 做 embedding 是否好用，<u>幽默的犀牛</u> 推荐了 Qwen 0.6B，表示「能明显感觉出比 grep 强」但 grep 找代码可能更高效。<u>今天群内信息量极大</u> 补充说 BGE 是上古时期的 embedding，现在推荐用 Qwen 的 embedding 或 OpenAI 的 embedding API，后者极其便宜，发几万个请求也就几毛钱。
