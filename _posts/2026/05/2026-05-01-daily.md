## 方法论

### Evaluations 怎么做：从文章到课程的讨论

<u>冷静的飞鼠</u> 邀请 <u>今天群内信息量极大</u> 手写一篇关于如何做 evaluations 及常见误区的文章。<u>今天群内信息量极大</u> 表示有兴趣但写作队列已经很长。<u>稳重的海豚</u> 认为「eval和benchmark太重要了，优化方向错了整个产品都得完蛋」。讨论延伸到是否应该做成系列课程，<u>冷静的飞鼠</u> 甚至提议「evals 十讲，$199」，<u>沉稳的兔子</u> 则认为这种内容「肯定是卖个案的，1M一次」。

<u>今天群内信息量极大</u> 指出核心难题在于抽象程度的平衡：「讲抽象了大家会觉得不落地，讲具体了别的领域的人听不懂。」<u>神秘的麋鹿</u> 从学习者角度分析了「道与术」的矛盾——抽象理论需要具体经验支撑才能被真正应用。<u>风趣的海豚</u> 提出 AI 时代反而适合更干的内容，细节可以让 AI 补充。

> <u>专注的剑鱼</u>：现在感觉除了基模团队，其他团队的eval连评估主体、评估维度都很混乱分散

> <u>稳重的海豚</u>：基模团队都不行，瞎搞

### Spec 审查与 AI 协作中的错误传播

<u>神秘的麋鹿</u> 坦言不太有耐心仔细看 AI 扩写的 PRD 和生成的 spec，只能看看关键路径。<u>80-HD</u> 分享了自己的做法：大概扫一眼，写 spec 最重要的是让 AI 自己搞清楚再干活；如果发现错误就 revert，告诉它哪里错了，重新生成 spec 从头做。

### P0 Gate 被模型跳过的问题

<u>严谨的长颈鹿</u> 提出一个实际问题：设置了 P0 gate（如必须询问语言才能进行下一步），但 5.5 等模型会自行 assume 已知信息跳过 gate，而 Opus 不会有这种情况，Cursor compress2 经常出现。

### 企业 Agent 架构：本地 vs 云端

<u>今天群内信息量极大</u> 围绕组内 share skill 话题提出了一个深度思考：个人用户的 context 在本地，local agent 有天然优势；但企业环境中 wiki、database、Slack 全在云端，cloud agent 可能更 make sense。关键问题是「这道线应该画在哪」。

> <u>今天群内信息量极大</u>：以前大家只是觉得 agent 的理想形式要么是云端，要么是本地。但具体到企业内部的 agent，我们也许应该问另一个问题：这道线应该画在哪？

<u>幽默的犀牛</u> 认为本地和云端会是动态平衡，通过 Glean MCP 等统一接口让本地 agent 接入云端数据，share skill 则让 agent 执行前读 Confluence 上的 runbooks。<u>细心的熊猫</u> 建议按任务类型区分，计算密集型留本地，云端做编排。<u>淡定的喜鹊</u> 分享了实际做共享 KB 的复杂性——既要让组员自主 contribute，又不能让 agent 互相瞎改核心知识，还需要 admin 做 quality control。<u>地球首帅鸭哥</u> 则主张让 shared memory 中的知识在实践中被检验、自动去掉错误的，<u>淡定的喜鹊</u> 反驳说面向人类 audience 时试错成本可能很高。

### 用文言文压缩 Persistent Memory

从中文 token 成本话题出发，<u>淡定的喜鹊</u> 表示真的想过用文言文存 persistent memory 来增加信息密度。<u>直率的狮子</u> 指出已有人做了类似项目：[caveman 的文言文模式](https://github.com/JuliusBrussee/caveman?tab=readme-ov-file#文言文-wenyan-mode)。
